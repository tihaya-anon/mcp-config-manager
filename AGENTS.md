# Repository Guidelines

## Project Structure & Module Organization
- Source code lives in `src/`, with extension entrypoint at `src/extension.ts`.
- MCP domain logic is under `src/mcp/` (types, serialization, store, tree, studio, exporter, templates).
- Built output goes to `out/` and is generated by TypeScript; do not edit `out/` manually.
- Static assets (for example activity bar icon) live in `resources/`.
- Root configs: `package.json` (scripts/contributions), `tsconfig.json` (strict TS config), `.vscode/` (workspace settings).

## Build, Test, and Development Commands
- `npm install`: install dependencies.
- `npm run compile`: compile `src/` to `out/` using `tsc`.
- `npm run watch`: incremental TypeScript compile during development.
- `npm run lint`: run ESLint on `src/**/*.ts`.
- `npm test`: runs `pretest` (`compile` + `lint`) then executes extension tests via `node ./out/test/runTest.js`.
- In VS Code, press `F5` to launch an Extension Development Host for manual testing.

## Coding Style & Naming Conventions
- Language: TypeScript with `strict: true`; keep types explicit for public/internal APIs in `src/mcp/`.
- Indentation: 2 spaces; keep imports grouped and consistent with existing files.
- Naming: `camelCase` for variables/functions, `PascalCase` for types/interfaces, kebab-case only for filenames where already established.
- Keep modules focused: add MCP logic to `src/mcp/*` rather than inflating `extension.ts`.
- Run `npm run lint` before opening a PR.

## Testing Guidelines
- Add/maintain extension tests under the existing test harness expected at `out/test/runTest.js`.
- Co-locate test source with the existing test layout (if adding new tests, mirror source module intent and naming).
- Minimum bar for changes: `npm run compile`, `npm run lint`, and `npm test` must pass locally.

## Commit & Pull Request Guidelines
- Follow the project’s commit style seen in history: Conventional Commit prefixes like `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`.
- Keep commits scoped and descriptive (example: `feat: add codex export path toggle`).
- PRs should include:
  - What changed and why.
  - Linked issue (if applicable).
  - Manual verification steps (commands run, Extension Host flows tested).
  - UI screenshots/GIFs for view or webview changes.

## Security & Configuration Tips
- Never commit secrets in MCP server `env` values or workspace settings.
- Prefer workspace-scoped test configs and sanitize examples in docs/PR descriptions.
